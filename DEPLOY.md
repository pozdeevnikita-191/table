# Деплой на Timeweb Cloud

Проект собирается в **один Docker-образ**: Express-сервер бэкенда отдаёт и
API (`/api/*`), и статику собранного фронтенда (всё остальное). Это упрощает
деплой — нужен всего один сервис, один порт, без CORS и без отдельного
фронтенд-приложения.

Есть два разумных способа развернуть его на Timeweb Cloud:

- **Вариант А (рекомендуется): App Platform + Dockerfile + управляемая
  PostgreSQL (DBaaS)** — меньше всего забот об инфраструктуре, автодеплой по
  git push, БД с бэкапами из коробки.
- **Вариант Б: обычный облачный сервер (VDS) + Docker Compose** — вы сами
  управляете сервером, Postgres поднимается в соседнем контейнере с
  постоянным диском (volume).

> Почему не Docker Compose прямо в App Platform? В App Platform запрещены
> директивы `volumes` в `docker-compose.yml` — то есть Postgres в контейнере
> там не сможет хранить данные между рестартами. Поэтому в App Platform Postgres
> должен быть отдельной управляемой БД, а не сервисом в compose-файле.

## Вариант А — App Platform (Dockerfile) + управляемая PostgreSQL

### 1. Создайте базу данных

В панели Timeweb Cloud: **Облачные базы данных → Создать кластер → PostgreSQL**.
После создания скопируйте строку подключения (вкладка «Подключение к БД») —
она понадобится как `DATABASE_URL`.

### 2. Залейте код в git-репозиторий

App Platform разворачивает приложения из GitHub/GitLab/Bitbucket (или по
прямой ссылке на git-репозиторий). Запушьте этот проект (со всеми файлами —
`Dockerfile`, `.dockerignore` и т.д. — в корне) в свой репозиторий.

### 3. Создайте приложение

В панели: **App Platform → Создать приложение → Docker → Деплой из
Dockerfile**. Подключите репозиторий и ветку.

Timeweb соберёт образ по корневому `Dockerfile` — он уже собирает финальный
(рантайм) слой по умолчанию, ничего дополнительно указывать не нужно.

### 4. Переменные окружения

В настройках приложения добавьте:

| Переменная     | Значение                                              |
| -------------- | ------------------------------------------------------ |
| `DATABASE_URL` | строка подключения к вашей PostgreSQL из шага 1        |
| `PORT`         | `3000` (можно не задавать — уже задан по умолчанию в образе) |

### 5. Путь проверки состояния

В настройках приложения укажите health-check путь: **`/api/healthz`**.

### 6. Запустите деплой

Нажмите «Запустить деплой» и дождитесь окончания сборки. По завершении
приложение будет доступно по бесплатному техническому домену (или своему
домену, если привяжете).

### 7. Примените схему БД (один раз, и после каждого изменения схемы)

Схема применяется командой `drizzle-kit push` — она сравнивает
`lib/db/src/schema` с БД и обновляет её. Проще всего сделать это со своей
машины, указав `DATABASE_URL` вашей БД в Timeweb (если у кластера включён
доступ по публичному IP):

```bash
corepack enable
pnpm install
DATABASE_URL="<строка подключения из шага 1>" pnpm --filter @workspace/db run push
```

Либо через сборку целевого слоя `migrate` из Dockerfile локально:

```bash
docker build --target migrate -t tabele-migrate .
docker run --rm -e DATABASE_URL="<строка подключения из шага 1>" tabele-migrate
```

## Вариант Б — обычный сервер (VDS/VPS) + Docker Compose

Подходит, если хотите держать и приложение, и базу данных на одном
арендованном сервере Timeweb Cloud.

### 1. Закажите сервер

**Облачные серверы → Создать сервер**, любой образ с Docker (или обычный
Ubuntu — Docker поставите сами):

```bash
curl -fsSL https://get.docker.com | sh
```

### 2. Скопируйте проект на сервер

```bash
git clone <ваш-репозиторий> tabele
cd tabele
cp .env.example .env
# отредактируйте .env — задайте свой POSTGRES_PASSWORD и т.д.
```

### 3. Поднимите приложение

```bash
docker compose up -d --build
```

Поднимутся два сервиса:

- `app` — Node.js-сервер (API + фронтенд), слушает порт из `PORT` (по
  умолчанию `3000`);
- `db` — PostgreSQL 16 с постоянным диском (`docker volume`).

### 4. Примените схему БД (один раз, и после каждого изменения схемы)

```bash
docker compose run --rm migrate
```

### 5. Откройте наружу

Проще всего поставить перед приложением обратный прокси (Nginx или Caddy) на
80/443 порту хоста, который проксирует на `127.0.0.1:3000`, и выпустить туда
TLS-сертификат (например, через Certbot). Это отдельный шаг настройки сервера,
не связанный напрямую с самим приложением.

### Обновление после изменений в коде

```bash
git pull
docker compose up -d --build
```

## Что было изменено в проекте для деплоя на Timeweb Cloud

- Express-сервер (`artifacts/api-server`) теперь в продакшене отдаёт ещё и
  статику фронтенда (`STATIC_DIR`) с SPA-фолбэком — один процесс вместо двух.
- Из `artifacts/tabele/vite.config.ts` убраны Replit-плагины
  (`@replit/vite-plugin-*`) и жёсткое требование переменных `PORT`/`BASE_PATH`
  — теперь у них есть безопасные значения по умолчанию, а `base` всегда `/`,
  так как фронтенд и API живут на одном origin.
- Добавлены `Dockerfile` (многоступенчатая сборка: install → build → лёгкий
  рантайм-образ без исходников и `node_modules`), `.dockerignore`,
  `docker-compose.yml`, `.env.example`.
- Удалены файлы, специфичные для Replit (`.replit`, Replit-плагины Vite,
  `replit.md` заменён на `README.md` + этот файл).
- `artifacts/mockup-sandbox` (дев-песочница для превью макетов) в сборку
  Docker не входит — он не нужен в продакшене.
